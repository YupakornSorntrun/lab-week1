# ปฏิบัติการสัปดาห์ที่ 2 : เปรียบเทียบ REST และ GraphQL พร้อมตัวอย่าง CRUD

## วัตถุประสงค์ของปฏิบัติการ

เมื่อทำปฏิบัติการนี้เสร็จสิ้น ผู้เรียนจะสามารถ

1. สร้าง endpoint แบบ REST และแบบ GraphQL สำหรับข้อมูลชุดเดียวกันได้
2. เปรียบเทียบจำนวนคำขอ (request) และปริมาณข้อมูลที่ได้รับกลับระหว่าง REST และ GraphQL ได้ด้วยตนเอง
3. เขียน GraphQL Query เพื่อเลือกเฉพาะฟิลด์ข้อมูลที่ต้องการได้
4. อธิบายปัญหา over-fetching และ under-fetching จากผลการทดลองจริงที่ตนเองสร้างขึ้น
5. เขียน GraphQL Query ที่มีการกรอง (filter), การเรียงลำดับ (sort), อาร์กิวเมนต์ (argument) และการดึงข้อมูลแบบซ้อน (nested query) ได้หลากหลายรูปแบบ
6. เขียน GraphQL Mutation เพื่อสร้าง แก้ไข และลบข้อมูลได้ครบตามหลักการ CRUD



## ผลการทดลองเปรียบเทียบ 
ภาพหน้าจอหรือบันทึกผลลัพธ์จาก

1. **REST endpoint `/api/v1/students/1/full` (แสดงปัญหา over-fetching)**

ทดสอบ `GET http://localhost:3000/api/v1/students/1/full` ด้วย Postman แล้วสังเกตผลลัพธ์



ผลลัพธ์ที่ได้รับกลับ

![รูปขั้นตอนที่2.4over-fetching](/images/2.3%20(REST-Over-fetching).png)

* **ข้อสังเกต (Over-fetching):** สมมติว่าหน้าจอที่ต้องการแสดงผลต้องการเพียง `name` และรายชื่อ `courseName` เท่านั้น 

* แต่ระบบ REST ข้างต้นคืนค่าฟิลด์อื่นที่ไม่จำเป็น เช่น `email`, `phone`, `courseIds` มาด้วยทุกครั้ง นี่คือปัญหา over-fetching 

---


2. **GraphQL Query ที่ขอเฉพาะ `name` และ `courses.courseName` (ขั้นตอนที่ 3.6 ใน `wk02-lab.md`)**

ขั้นตอนที่ 3.6 ทดลองเขียน Query เพื่อแก้ปัญหา Over-fetching

ในหน้า GraphiQL ทดลองพิมพ์ Query ต่อไปนี้ เพื่อขอเฉพาะ `name` และชื่อรายวิชา โดยไม่ขอฟิลด์อื่นที่ไม่จำเป็น

```graphql
query {
  student(id: 1) {
    name
    courses {
      courseName
    }
  }
}
```

ผลลัพธ์ที่ได้รับกลับ

![รูปขั้นตอนที่3.6](/images/3.6.png)

---

3. **GraphQL Query ที่รวมข้อมูลนักศึกษาทุกคนพร้อมรายวิชาในคำขอเดียว (ขั้นตอนที่ 3.7)**

ขั้นตอนที่ 3.7 ทดลองเขียน Query เพื่อแก้ปัญหา Under-fetching

ทดลอง Query ที่รวมข้อมูลจากหลาย resource ไว้ในคำขอเดียว

```graphql
query {
  students {
    name
    major
    courses {
      courseName
      credit
    }
  }
}
```
ผลลัพธ์ที่ได้รับกลับ

![รูปขั้นตอนที่3.7(1)](/images/3.7(1).png)
![รูปขั้นตอนที่3.7(2)](/images/3.7(2).png)

* **เปรียบเทียบกับข้อ 1 REST endpoint `/api/v1/students/1/full`** 
* สังเกตว่า Query เดียวนี้ให้ข้อมูลนักศึกษาทุกคนพร้อมรายวิชาที่ลงทะเบียนครบถ้วน โดยใช้เพียง 1 คำขอ ในขณะที่แนวทาง REST เดิมต้องใช้หลายคำขอเพื่อให้ได้ข้อมูลชุดเดียวกัน

## **ตารางสรุปผลการเปรียบเทียบ** 
ตามรูปแบบในขั้นตอนที่ 3.8 ของ `wk02-lab.md` (จำนวนคำขอ และฟิลด์ที่ได้รับเกินความจำเป็น)

| หัวข้อเปรียบเทียบ                                    | REST (`/students/:id/full`)               | GraphQL                          |
| ---------------------------------------------------- | ----------------------------------------- | -------------------------------- |
| จำนวนคำขอที่ใช้เพื่อดูชื่อและรายวิชาของนักศึกษา 1 คน | 1 คำขอ (แต่ได้ฟิลด์เกินความจำเป็น)        | 1 คำขอ (ได้เฉพาะฟิลด์ที่ต้องการ) |
| จำนวนคำขอที่ใช้เพื่อดูชื่อและรายวิชาของนักศึกษาทุกคน | มากกว่า 1 คำขอ หากต้องแยกดึงข้อมูลรายวิชา | 1 คำขอ                           |
| ฟิลด์ที่ได้รับเกินความจำเป็น (over-fetching)         | มี เช่น `email`, `phone`, `courseIds`     | ไม่มี                            |
| การเลือกฟิลด์ข้อมูล | ไม่สามารถเลือกเฉพาะฟิลด์ได้ Endpoint  ส่งข้อมูลที่กำหนดไว้ทั้งหมด | เลือกเฉพาะฟิลด์ที่ต้องการได้ เช่น `name` และ `courses.courseName` |
ความยืดหยุ่นของการเรียกข้อมูล | ต่ำ ต้องแก้ไข Endpoint ถ้าต้องการข้อมูลรูปแบบใหม่/ข้อมูลอื่นๆ | สูง เราสามารถกำหนดข้อมูลที่ต้องการผ่าน Query ได้เอง |
